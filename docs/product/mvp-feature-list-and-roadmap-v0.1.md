# MVP 功能清单与开发排期 v0.1

更新时间：2026-03-12

## 1. 文档目标

这份文档用于回答两个问题：

1. 第一版到底做哪些功能。
2. 这些功能按什么节奏推进最稳。

默认前提：

- 桌面优先
- 单用户优先
- 本地优先
- Excalidraw 作为左侧来源画布引擎
- 应用支持后台常驻
- 架构按未来会上后端来设计

## 2. 第一版要验证的核心假设

第一版不是要验证“AI 很强”或“无限画布很酷”，而是要验证这三个假设：

1. 用户愿不愿意在不打开主窗口的情况下，用全局快捷键把信息先收进左侧画布。
2. 用户会不会觉得右侧待办投影比手工录入更轻松。
3. 用户在 hover / click 联动时，是否真的感到“上下文没丢”。

## 3. MVP 功能清单

### 3.1 P0：必须成立的主闭环

| 模块 | 功能 | 说明 | 验收标准 |
| --- | --- | --- | --- |
| 桌面壳 | 托盘常驻 | 关闭主窗口后仍可驻留后台 | 应用可在后台运行并被唤起 |
| 桌面壳 | 全局快捷键 | 从任意应用触发收录 | 用户可在桌面全局触发收录 |
| 捕获 | 剪贴板文字读取 | 复制文本后可直接进入系统 | 文本能稳定保存 |
| 捕获 | 剪贴板图片读取 | 截图或复制图片后可直接进入系统 | 图片能稳定保存并渲染 |
| 反馈 | 系统轻量通知 | 在右下角提示“已记录/正在整理” | 不打开主窗口也能收到明确反馈 |
| 架构 | Repository 抽象层 | 业务层不直接依赖具体存储 | 本地实现可替换而业务层不改 |
| 架构 | Schema version | 数据模型支持未来迁移 | 所有核心对象都有 schemaVersion |
| 架构 | AI adapter 抽象 | AI 调用不写死供应商 | 可替换 provider 而不改业务流程 |
| 来源层 | 左侧主画布 | 承载原始信息和来源卡片 | 可平移、缩放、查看来源 |
| 投影层 | 右侧待办列表 | 展示从来源提取出的待办 | 列表能实时展示待办项 |
| 生成逻辑 | 来源卡片自动生成 | 捕获后在左侧落卡 | 新输入能自然落位 |
| AI | 摘要和待办建议 | 对新捕获做轻量提取 | AI 失败不影响捕获 |
| 关联 | 待办与来源映射 | 一个待办能回到对应来源 | 每条待办至少能定位一个来源 |
| 联动 | hover 高亮 + 连线 | 鼠标移到右侧时左侧高亮 | hover 延迟低且反馈明显 |
| 联动 | 点击定位来源 | 点击右侧待办自动平移到来源 | 用户能一键回到上下文 |
| 基础编辑 | Inspector | 查看原文、AI 提示、状态 | 可编辑标题/状态/说明 |
| 存储 | 本地优先持久化 | 重启后数据仍在 | 数据不丢失 |
| 搜索 | 基础搜索 | 按文本找来源或待办 | 可定位到相关项 |

### 3.2 P1：让产品开始好用

| 模块 | 功能 | 价值 |
| --- | --- | --- |
| AI | 时间线索提取 | 让待办更可执行 |
| AI | 多来源合并建议 | 避免同一件事拆成碎片 |
| 投影层 | 待办状态流转 | `待确认 / 待办 / 稍后 / 完成` |
| 联动 | 来源反向高亮待办 | 从左侧也能看到行动层 |
| 搜索 | 标签和状态筛选 | 让回看更高效 |
| 画布 | 默认弱结构分区 | 降低空白恐惧和混乱感 |

### 3.3 P2：提高留存与长期使用性

| 模块 | 功能 | 价值 |
| --- | --- | --- |
| 数据 | 去重与相似合并 | 控制垃圾堆积 |
| 组织 | 多 Board 管理 | 适应更长期使用 |
| 接入 | 浏览器扩展 | 强化网页场景捕获 |
| 同步 | 云同步 | 支持跨设备 |
| 智能 | 回顾建议 | 提醒长期未处理内容 |

## 4. 明确不做

第一版不做：

- 多人协作
- 分享链接
- 项目管理体系
- 重度提醒和重复任务
- 子任务树
- 移动端优先
- 复杂权限系统

这些功能都会稀释“来源到待办投影”这个核心闭环。

## 5. 研发拆分建议

建议按 5 个阶段推进。

### 阶段 0：产品壳和数据骨架

目标：

- 跑起来一个可工作的 Electron + React + Excalidraw 壳
- 跑起来一个可后台常驻的桌面壳
- 建立 Capture、SourceCard、TaskItem 的业务模型

交付：

- 项目骨架
- 托盘与主窗口生命周期骨架
- Repository 接口骨架
- AI adapter 接口骨架
- 本地数据存储骨架
- 左右双栏基础布局

### 阶段 1：捕获闭环

目标：

- 从复制到左侧画布出现来源卡片

交付：

- 全局快捷键
- 后台触发收录
- 剪贴板文字/图片读取
- 系统轻量通知
- 左侧来源卡片生成
- 本地持久化

### 阶段 2：待办投影闭环

目标：

- 从来源卡片到右侧待办列表自动生成

交付：

- AI / 规则待办提取
- 右侧待办列表
- 待办状态基础模型

### 阶段 3：联动体验闭环

目标：

- 让双栏模型真正有识别度

交付：

- hover 高亮
- 临时连线
- 点击定位来源
- 来源反向高亮待办

### 阶段 4：好用度补强

目标：

- 让产品从“可演示”变成“可连用”

交付：

- Inspector
- 搜索
- 状态筛选
- 默认画布弱结构

## 6. 建议排期

这是一个偏保守、适合单人或小团队的 8 周版本。

| 周期 | 目标 | 输出 |
| --- | --- | --- |
| 第 1 周 | 项目骨架 | Electron 壳、托盘常驻、前端骨架、数据模型初版、Repository 与 AI adapter 骨架 |
| 第 2 周 | 捕获能力 | 全局快捷键、后台触发、剪贴板读取、系统通知、本地存储 |
| 第 3 周 | 左侧来源画布 | 来源卡片生成、自动落位、基础画布操作 |
| 第 4 周 | 右侧待办投影 | AI/规则提取、待办列表初版 |
| 第 5 周 | 双栏联动 | hover 高亮、点击定位、关联模型 |
| 第 6 周 | Inspector 与编辑 | 原文查看、状态编辑、基础搜索 |
| 第 7 周 | 稳定性与默认布局 | 画布弱结构、错误处理、性能修补 |
| 第 8 周 | 内测准备 | 埋点、文案、首轮走查、体验修正 |

## 7. 里程碑定义

### M1：能收

标准：

- 用户复制文字或图片后能被成功收录
- 用户不需要先打开主窗口
- 左侧画布出现来源卡片

### M2：能投影

标准：

- 来源能稳定生成右侧待办
- 用户不需要自己重新打一遍待办

### M3：能回指

标准：

- 用户在右侧看待办时能快速回到左侧来源
- hover / click 体验自然、低负担

### M4：能连用

标准：

- 用户可以连续几天使用而不觉得完全失控
- 搜索和状态流转足以支撑回看

## 8. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 提取质量一般 | 待办投影不可信 | 第一版允许规则兜底和人工改写 |
| hover 过度花哨 | 干扰使用 | 连线做瞬时、轻量、默认短时出现 |
| 左侧画布太乱 | 新用户无从下手 | 先提供弱结构分区和自动落位 |
| 右侧列表太强 | 产品变成普通待办 | 强调所有待办都必须有来源映射 |
| 后台常驻不稳定 | 无法形成无感习惯 | 先把托盘、通知、快捷键稳定性放到 P0 |
| 过早把本地实现写死 | 后续云同步成本陡增 | 从 P0 就把存储和 AI 抽象层分开 |

## 9. 后端预留要求

虽然第一版不上后端，但当前阶段必须完成这些准备：

- 核心对象稳定 ID
- 持久化对象带 `schemaVersion`
- 附件与元数据分离
- Repository 抽象层
- AI provider 抽象层
- 为未来账户、同步、搜索索引预留字段

## 10. 当前建议

最值得坚持的优先级顺序：

1. `先把收录做顺`
2. `再把待办投影做准`
3. `最后把双栏联动做漂亮`

如果顺序反过来，很容易做成一个演示很酷但不愿意长期使用的产品。
